# 2.4 Gate Enforcement [DONE]

## Description
Structural gate checks that validate phase transitions. When the wrapper
detects a card's status has changed, it runs the appropriate gate check
before accepting the transition. The system has three layers:

1. **Section parser** — extracts artifact sections (Decision, Contracts,
   Threshold Registry, Behavioral Spec, Cross-Cutting Note) from card
   markdown via `parseCardSections(rawContent)`.
2. **Gate functions** — four transition-specific checks that use the parser
   to validate section presence and structure:
   - PLAN → ARCHITECT: Description + AC present
   - ARCHITECT → IMPLEMENT: File Manifest + keyword-triggered sections
   - IMPLEMENT → REVIEW: all manifest files exist on disk
   - REVIEW → DONE: no open regressions, AC present
3. **Wrapper integration** — `runGateForTransition` dispatcher called after
   each transition, with advisory (log + continue) or blocking (revert) mode
   controlled by `WrapperConfig.gateMode`.

Reference: `docs/taxonomy-v2-specification.md` §2.9.

## Decision

### Context
The gate system needs to parse card sections, check structural requirements,
and integrate with the wrapper loop. Key choices: how to represent parsed
sections, how to organize gate functions, and how to dispatch from the wrapper.

### Options Considered
1. **Flat boolean flags** for section detection vs **single structured result object**
2. **Single dispatcher** for gates vs **four named functions** with a dispatcher on top
3. **Inline switch in wrapper** vs **gate dispatcher function** in gate-checks.ts

### Choice
- Single `SectionParseResult` object from one `parseCardSections()` call
- Four named gate functions (`checkPlanToArchitect`, `checkArchitectToImplement`,
  `checkImplementToReview`, `checkReviewToDone`) plus a `runGateForTransition`
  dispatcher that maps `(from, to)` to the correct function
- Dispatcher lives in `gate-checks.ts`; wrapper calls it with one line

### Rationale
- Single result object: minimal API surface, no redundant re-parsing
- Named functions: each has distinct signature (only IMPLEMENT→REVIEW needs
  filesystem), independently testable
- Dispatcher in gate-checks.ts: keeps wrapper clean, new gates added without
  touching wrapper

### Consequences
- Positive: All gate logic centralized in gate-checks.ts
- Positive: Wrapper changes minimal — one dispatcher call + react to result
- Trade-off: Parser does full extraction even if consumer needs one section,
  but card markdown is small enough (~100-500 lines) to be negligible

## Contracts

### Preconditions
- `parseCardSections` receives a string (full card markdown including YAML frontmatter)
- Gate functions receive a `Card` object with `rawContent` and `fileManifest` populated
- `checkImplementToReview` additionally receives `existsSync: (path: string) => boolean`
- `runGateForTransition` receives valid `LeafPhase` values for `from` and `to`
- `WrapperConfig.gateMode` is `"advisory"` (default) or `"blocking"`

### Postconditions
- `parseCardSections` returns `SectionParseResult` with accurate presence flags; never throws
- Gate functions return `GateResult` with `pass: true` iff `violations.length === 0`; never throw
- `runGateForTransition` returns `{ pass: true, violations: [] }` for unregistered transitions
- Advisory mode: transition kept, violations logged as warnings, context passed to next prompt
- Blocking mode: `beforeContent` written back to card file, violations logged as errors

### Invariants
- Gate checking only runs when a phase transition is detected
- Section detection is case-insensitive, regex/string-based (no AST)
- Subsection detection scoped within parent section's content range
- Parser has no side effects; only `checkImplementToReview` touches filesystem (via injected fn)
- Advisory never prevents a transition; blocking always reverts a failed one
- `gateContext` cleared after consumption in next iteration's prompt

## Behavioral Spec

### Scenario 1: Advisory mode — gate passes
Given a card transitioning PLAN→ARCHITECT in advisory mode
When the gate check passes (no violations)
Then the transition is accepted, no warnings logged

### Scenario 2: Advisory mode — gate fails
Given a card transitioning PLAN→ARCHITECT in advisory mode
When the gate check returns violations (e.g., missing Description)
Then the transition is kept, violations logged as warnings, next prompt includes gate context

### Scenario 3: Blocking mode — gate fails and reverts
Given a card transitioning PLAN→ARCHITECT in blocking mode
When the gate check returns violations
Then the card file is reverted to pre-transition content, violations logged as errors

### Scenario 4: No transition — no gate check
Given a card modified but status unchanged
Then no gate check runs

### Scenario 5: Unknown transition — pass through
Given a transition with no registered gate (e.g., node PLAN→DONE)
Then `runGateForTransition` returns `{ pass: true, violations: [] }`

### Scenario 6: Gate context in system prompt
Given gate violations from previous iteration
When `generateSystemPrompt` called with `gateContext`
Then prompt includes `## Gate Violations from Previous Iteration` section
