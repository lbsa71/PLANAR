# 5.3 Card Invalidation [IMPLEMENT]

## Description
Regress affected cards to the appropriate phase based on their current status.
Every regression includes a content change (upstream diff summary) satisfying
the Challenge rules. This is the final step of the git watch cycle — after
impact analysis identifies which cards are affected by upstream changes, this
module performs the actual card file modifications.

See @README.md — "Git Watch Mode: How It Works" step 7 and "Design Principles".

## File Manifest
- src/invalidation.ts
- src/invalidation.test.ts
- src/card.ts (read — uses parseCard, updateCardStatus, isPhase)
- src/types.ts (read — uses Card, CardStatus, LeafPhase, FileSystem)
- docs/5-git-watch/ARCHITECTURE.md

## Acceptance Criteria
- Receives a list of affected cards (from 5.2 impact analysis) with changed file details
- Cards in [PLAN]: left alone (planning hasn't committed to anything)
- Cards in [ARCHITECT] or later (but not [DONE]): regressed to [PLAN] with upstream change note
- Cards in [DONE]: regressed to [REVIEW] (implementation may still be valid)
- Content change included in every regression: appends note like "Upstream changes in [files] may affect this card's assumptions. Re-evaluate."
- No silent invalidation — every regression explains why
- Exported as a function callable from the git watch loop
- Reads and rewrites card files using existing card.ts parsing
